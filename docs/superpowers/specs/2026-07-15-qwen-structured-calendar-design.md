# Qwen 结构化校历入口设计

## 目标

为现有校历模块增加一个由 Qwen API Key 开启的结构化展示入口：

1. 设置页允许用户填写、保存和清除 Qwen API Key。
2. 未配置 Key 时，校历页继续显示教务处原始图片，并在页面角落提示用户可在设置中开启智能解析。
3. 已配置 Key 时，每次进入校历页显示 `thinking...`，随机等待 20–60 秒后展示当前所选学年的结构化校历。
4. 学年选择器和刷新按钮在图片模式、等待状态和结构化模式中始终保留并可用。
5. 预先解析四川大学教务处校历列表当前发布的全部数据，并以版本化结构化资源随源码发布。

## 数据范围

数据基线为 2026-07-15 的 `https://jwc.scu.edu.cn/cdxl.htm`。该页面当前包含以下 6 个学年：

- 2026—2027
- 2025—2026
- 2024—2025
- 2023—2024
- 2022—2023
- 2021—2022

每个学年包含秋季、春季两个语义学期，共 12 个学期。2022—2023 详情页同时包含两套不同分辨率的秋、春季图片；结构化资源按语义去重，只保存两个学期，不把分辨率副本视为额外数据。

本次“全部数据”包含图片中能够结构化表达的：

- 学年和学期标题；
- 每个周次的起止日期、周次编号和阶段；
- 报到、补缓考、开学典礼、正式行课、集中考试、实践周、寒暑假等日期或日期区间；
- 节假日及图片中以红色标出的停课日期；
- 图片右侧全部备注原文；
- 原列表页、详情页、发布日期和原图 URL，供来源对应和核验。

## 方案选择

采用“版本化 JSON 资源 + C++ 强校验加载器 + QML 结构化视图”。

未采用以下方案：

- 将全部数据硬编码在 QML/JavaScript 中：实现虽短，但日期校验、模型复用和测试能力差。
- 运行时 OCR 或调用真实模型：不符合预先解析并把全部结构化数据存入源码的要求。
- 为每学年建立独立数据库表：数据是随版本发布的只读内容，数据库迁移和写入层没有收益。

## 结构化资源

新增 `SCU_Nexus/resources/calendar/academic_calendars.json`，顶层格式如下：

```json
{
  "schemaVersion": 1,
  "sourceIndexUrl": "https://jwc.scu.edu.cn/cdxl.htm",
  "capturedAt": "2026-07-15",
  "calendars": []
}
```

每个学年文档使用以下字段：

```json
{
  "academicYear": "2026-2027",
  "title": "2026—2027学年四川大学校历",
  "sourcePagePath": "/info/1101/10727.htm",
  "publishedAt": "2026-07-08",
  "terms": [
    {
      "id": "2026-fall",
      "name": "秋季学期",
      "startDate": "2026-08-27",
      "endDate": "2027-02-27",
      "sourceImageUrl": "https://jwc.scu.edu.cn/...",
      "weeks": [
        {
          "weekNo": 1,
          "phase": "teaching",
          "startDate": "2026-08-30",
          "endDate": "2026-09-05",
          "label": "教学周"
        }
      ],
      "events": [
        {
          "id": "student-registration",
          "type": "registration",
          "title": "在校生报到",
          "startDate": "2026-08-27",
          "endDate": "2026-08-28",
          "affectsTeaching": false
        }
      ],
      "notes": [
        {
          "order": 1,
          "text": "8月27-28日为在校生报到时间。"
        }
      ]
    }
  ]
}
```

稳定枚举值如下：

- `phase`：`registration`、`teaching`、`exam`、`practice`、`winter_break`、`summer_break`、`other`。
- `event.type`：`registration`、`makeup_exam`、`orientation`、`class_start`、`holiday`、`exam`、`practice`、`ceremony`、`sports_meeting`、`other`。

日期统一使用 ISO 8601。尚未进入正式周次的报到日期不伪造“第 0 周”，只作为事件保存。备注保留图片原文；程序逻辑使用规范化事件和阶段，不解析备注文本。

## C++ 数据边界

扩展 `AcademicCalendarModels`，增加结构化学年、学期、周、事件和备注类型。JSON 只承担资源序列化，QML 不直接操作未校验的 `QJsonObject`。

新增 `AcademicCalendarCatalog`，职责为：

- 从 Qt 资源读取唯一的结构化 JSON；
- 严格校验 schema 版本、6 个学年、每学年两个学期、日期格式、日期顺序、周次唯一性、周日期连续性、事件区间和来源字段；
- 按现有条目的标题或详情页 path 查找对应学年；
- 向 ViewModel 返回强类型结构和 QML 可消费的 `QVariantMap`；
- 结构化资源缺失或损坏时返回明确错误，不影响现有图片加载链路。

`AcademicCalendarViewModel` 保留现有条目、选择、刷新、图片缓存和网络抓取职责，并增加只读属性：

- `structuredCalendar`：当前学年的结构化数据；
- `hasStructuredCalendar`：当前学年是否有内置结构化数据；
- `structuredCalendarError`：本地资源错误信息。

选中学年变化时同步切换结构化数据，但不改变或取消原有详情请求。刷新操作继续刷新官网条目和图片；结构化资源保持只读。

## API Key 设置

`AppSettings` 增加 `qwenApiKey`、`hasQwenApiKey`、保存和清除接口，并通过现有 `QSettings` 持久化。Key 输入框使用密码回显模式；设置页只显示 Qwen API Key 的用途、保存状态和操作按钮，不出现“仅用于演示”“不会发送”或同义提示。

Key 只作为是否开启结构化模式的开关，不新增 Qwen 网络客户端、请求 DTO 或 API 调用。保存或清除后发出属性变更信号，使下一次进入校历页读取到最新状态。

## 校历页面状态

页面保持现有标题栏、学年选择器和刷新按钮。内容区域增加三种互斥显示状态：

### 图片模式

当 `appSettings.hasQwenApiKey` 为 false 时，继续使用现有 `imageUrls`、图片加载状态和大图查看器。页面右下角显示不遮挡主要内容的智能解析提示；点击后跳转设置页。

### 等待状态

当 `appSettings.hasQwenApiKey` 为 true 时，页面创建完成即生成 `[20000, 60000]` 毫秒闭区间内的随机时长并启动单次 Timer。内容区域显示 BusyIndicator 和精确文案 `thinking...`。

等待期间：

- 学年选择器保持可用；
- 刷新按钮保持可用并继续执行原有刷新；
- 用户切换学年时只更新待展示的目标学年，不重新计时；
- 页面销毁时 Timer 自动销毁，重新进入页面重新随机等待。

### 结构化模式

Timer 到期后显示当前所选学年的结构化数据。切换学年立即显示对应内置数据，不再次等待。结构化视图包含：

- 秋季/春季学期切换；
- 学期起止日期和阶段摘要；
- 按周次展示的起止日期、周类型和当前周强调；
- 按时间排序的事件列表；
- 可展开的原始备注列表；
- “查看原版”入口，继续复用现有图片查看能力。

若官网未来出现尚未收录的新学年，或本地资源校验失败，Timer 到期后回退图片模式并显示“当前学年暂未完成智能解析”，学年选择和刷新仍不隐藏。

## 错误与边界行为

- 空白或纯空格 Key 等同于未配置。
- 保存 Key 时去除首尾空格；清除后下次进入校历页恢复图片模式。
- 图片网络请求失败不阻止已经内置的结构化校历展示。
- 本地结构化资源损坏不影响图片模式和现有缓存。
- 等待状态不会发起额外网络请求，也不会阻塞现有校历列表刷新。
- 提示文案不包含“演示”“模拟”“不会发送”“仅用于”等描述。

## 测试与验收

实施采用测试驱动流程，每个生产代码步骤前先添加并运行会因功能缺失而失败的测试。

### 数据测试

- JSON 顶层 schema 为 1，来源 URL 和采集日期正确。
- 恰好包含基线页面的 6 个学年和 12 个语义学期。
- 每学年秋、春学期各一个，详情页 path 和发布日期与列表一致。
- 周次、阶段、起止日期、事件和备注均非空且合法。
- 周日期连续、每周 7 天、同一学期周次不重复、事件区间不反向。
- 2022—2023 不因重复分辨率图片产生四个学期。

### 设置测试

- Key 保存后可从新的 `AppSettings` 实例恢复。
- 首尾空格被移除，空白值不启用功能。
- 清除 Key 后持久化值和 `hasQwenApiKey` 同步更新。

### QML 契约测试

- 设置页包含密码输入、保存和清除操作。
- 校历页包含 `thinking...` 和 20–60 秒随机区间。
- 无 Key 提示可导航设置页。
- 学年选择器和刷新按钮不受三种内容状态的 `visible` 条件控制。
- 页面和新增组件不包含被禁止的提示词。
- 结构化视图包含学期、周次、事件、备注和原版图片入口。

### 最终验证

- 构建应用目标和全部测试目标。
- 运行新增聚焦测试并确认红绿过程。
- 运行完整 CTest 测试集。
- 检查 6 个学年数据与 12 张语义校历图片逐项对应。
- 检查提交历史中每个完成步骤均有独立提交。

## 提交边界

按以下独立、可验证结果提交：

1. 设计说明。
2. 实施计划。
3. 全部结构化校历资源及完整性测试。
4. API Key 持久化及设置页入口。
5. 结构化数据模型、加载器和 ViewModel 接口。
6. 结构化校历 QML 视图。
7. `thinking...` 等待状态、无 Key 引导和模式切换。
8. 最终验证中发现的必要修正。

## 非目标

- 不新增真实 Qwen API 请求。
- 不移除或替换现有教务处校历抓取、图片缓存和大图查看逻辑。
- 不隐藏学年选择器或刷新按钮。
- 不自动抓取未来新发布学年并写回只读资源。
- 不把重复分辨率图片保存为重复学期。
