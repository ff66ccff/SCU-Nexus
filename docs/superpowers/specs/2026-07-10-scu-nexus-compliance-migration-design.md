## 1. 背景与目标

本轮工作的目标不是将参考项目整体翻译成 Qt，而是建立一条可验证的需求闭环：逐条核对当前 Qt 项目是否满足项目文档，对缺失或行为不一致的 P0/P1 能力进行补齐，并从参考项目迁移经过验证、且符合当前分层架构的业务语义。

最终结果必须同时回答三个问题：

1. 每条正式需求和验收项在当前项目中是否已实现；
2. 如果已实现，代码、测试和运行证据分别在哪里；
3. 如果未实现，本轮如何补齐，或为何必须标记为外部人工验收。

明确标注为 P2、第二阶段或扩展能力的功能不因参考项目已经实现而自动进入本轮范围。

### 3.2 明确不在本轮实现

- REQ-013 对应的真实 P2/第二阶段业务功能；

真实账号和干净 Windows 环境属于外部条件。本轮提供可执行脚本、检查清单和记录位置，但不伪造通过证据。

基线说明项目不是从零实现。后续修改遵循“证据确认后只补缺口”的原则，避免复制已经存在的逻辑。

## 6. 总体设计

### 6.1 架构边界

继续采用现有分层：

```text
QML 页面与通用组件
        ↓ 仅调用属性、信号和 Q_INVOKABLE
ViewModel / AppController
        ↓
Service / Repository / QueryCache
        ↓
Qt Network、JSON、文件与本地设置
```

必须维持以下约束：

- QML 中不出现业务 URL、SQL、Cookie 操作、密码加密或 SM2 实现；
- 页面不直接持有网络回复对象或数据库连接；
- ViewModel 负责页面状态与用户可读错误，Service 负责协议和解析，Repository 负责持久化；
- 参考项目的代码只能迁移业务规则，不逐行翻译 UI 或状态管理框架；
- 所有跨层依赖由静态脚本检查，避免后续回归。

### 6.2 查询与缓存状态

统一查询状态扩展为：

```text
Idle → Loading → Loaded / Empty / Error / LoginRequired
Loaded / Empty → Refreshing → Loaded / Empty
Loaded / Empty → Refreshing → 保留原数据，并提示刷新失败
```

`Refreshing` 与首次 `Loading` 必须分离：首次加载可以显示骨架或加载页；已有缓存时刷新不得遮蔽内容。刷新失败时保留旧数据和原时间戳，使用非阻断提示反馈错误。

缓存读取失败不能被当作合法空数据。损坏 JSON、版本不兼容和持久化写入失败必须形成可诊断的错误，同时允许网络刷新恢复。

### 6.3 错误模型

统一错误类别至少覆盖：

- 网络不可达、超时、服务不可用、限流；
- 未认证、会话过期、凭据错误、验证码错误；
- 参数无效、冲突、数据写入失败；
- 响应解析失败、合法空结果、未知错误。

Service 返回稳定类别和安全摘要；ViewModel 将其映射为页面状态和可操作提示。合法空结果与解析失败必须区分，尤其是考试、成绩和校历 HTML/JSON 结构变化场景。

## 7. 分模块设计

### 7.1 A：应用框架与通用 UI

1. 实现 `AppSettings`，保存并恢复窗口大小、位置以及文档要求的持久设置；恢复时限制到可用屏幕，默认窗口为 1100×760，最小窗口为 900×620。
2. 让 `AppController::initialize()` 统一执行缓存与仓储初始化，并将失败转换为明确启动错误；入口程序不再忽略 `QueryCache::open()` 的结果。
3. 在设置页接入真实缓存清理 ViewModel，清理成功后更新状态和提示，失败时保留可诊断错误；移除“接口已预留”占位行为。
4. 将 `QueryStatePane` 的自定义 `state` 属性改为不覆盖 `QQuickItem::state` 的名称，并支持 `Refreshing`。
5. 修复 lint 指出的布局受管项仍设置宽高、未使用导入等确定性问题。大量由 C++ 上下文属性引起的静态提示单独记录，不通过全局禁用掩盖真实警告。
6. 清理已不参与导航的占位页资源和过时页面文案，但不做无关视觉重设计。
7. Toast 和启动日志不得输出未脱敏的用户输入。

### 7.2 B：认证、网络与教务接口

1. 保留当前统一认证、Cookie、SM2 和教务 API 主体，仅补安全和协议缺口。
2. 认证日志不再记录原始学号、用户名、密码、完整 token、Cookie 或成绩内容；扩展脱敏器及测试，日志只保留阶段、结果、错误类别和短请求标识。
3. 删除或收紧可返回完整 Cookie 值的调试接口，统一提供名称列表、数量或脱敏摘要。
4. 扩展统一网络错误枚举，使参数无效、冲突、数据写入失败和合法空结果不再依赖任意字符串约定。
5. API 解析必须区分“服务器明确返回空列表”和“页面结构变化导致未解析到数据”。考试计划、成绩和校历均使用这一规则。
6. 网络请求统一设置合理超时、User-Agent、HTTP 状态检查，并仅记录有限长度的脱敏响应摘要。
7. 继续保留当前明确记录为技术债的本地 token 存储方案；本轮不引入 Windows Credential Manager，但确保不存储明文密码。
8. 增加验证码字段缺失、Cookie 属性边界、解析失败和日志隐私等测试。

### 7.3 C：课表管理与教务导入

1. 从参考项目迁移旧课表 JSON 的兼容语义：识别旧 `semesterEndDate`、旧 `sectionsPerDay`，并在时间段缺失时生成有效默认值。
2. 修正 `Course::conflictsWith(excludeId)`：只排除正在检查的当前课程 ID，不因对方课程 ID 相等而静默跳过冲突；增加明确单测。
3. 将课程字段验证和冲突判断集中到共享业务规则。无论从编辑 ViewModel 还是公开 `ScheduleViewModel` 方法调用，都不能绕过课程名、星期、周次、节次和时间范围检查。
4. 仓储层的“替换课表课程”契约保证生成新的课程 ID，不能依赖某个调用者预先处理。
5. 未登录时，教务导入入口显示统一 `LoginRequired` 行为并可跳转登录；同步和导入不得只表现为无解释的禁用按钮或原始错误字符串。
6. 课表仓储在应用初始化阶段就绪，页面首次打开不承担全局初始化副作用。
7. 补充旧格式迁移、排除 ID、公开接口验证、导入登录态和替换 ID 的测试。

### 7.4 D：校历、考试与成绩

1. 校历服务接入共享网络策略：超时、User-Agent、HTTP 状态、字符集稳健解析和安全摘要；带引号或额外参数的 `charset` 仍能正确识别。
2. 校历解析为空时，只有服务端存在明确空结果语义才返回 Empty，否则返回 ParseFailed。
3. 校历刷新后按稳定标识恢复用户选中的条目；缓存损坏和写入失败显式上报；图片加载失败显示可理解的占位状态。
4. 校历、考试、成绩 ViewModel 使用 `Refreshing`，刷新中保留现有内容；清理缓存时同时重置时间戳、搜索词、选择项和过期错误状态。
5. 修正 `SchemeScoreSummary::fromJson()` 在“全部未通过”场景重复累计失败数的问题。
6. 成绩字段使用稳健的 JSON 标量转字符串逻辑，使数字类型的学分、原始成绩等字段不会丢失。
7. 方案成绩页面展示必修、选修等文档要求的学分汇总；分组标题显示通过数量和学分；课程卡展示原始成绩字段。
8. 自定义统计支持当前筛选结果全选/取消、按学期全选，并展示失败数量；选择状态与搜索和属性过滤保持一致。
9. 增加字符集、缓存损坏、刷新保留、登录态、空结果、排序、必修 GPA、筛选选择、全失败计数和数字字段解析测试。

### 7.5 E：测试、冲突检查与交付

1. 建立 `SCU_Nexus/tests/fixtures/`，存放不含真实隐私信息的认证、课表、校历、考试和成绩固定样本。
2. 为应用启动、导航、通用状态组件和设置页增加轻量自动化或 QML smoke test，覆盖 A 模块目前缺失的验证层。
3. 增加 `SCU_Nexus/scripts/conflict_check.py`，至少检查：QML 中业务 URL/SQL/Cookie/SM2、禁止的跨层 include、重复类型注册和明显占位实现。
4. 增加 `SCU_Nexus/scripts/package_windows.ps1`：执行 Release 构建、调用 `windeployqt`、复制 OpenSSL 等运行时依赖、生成发布目录和文件清单。
5. 增加 `SCU_Nexus/scripts/verify_package.ps1`：在不依赖源码目录和开发环境 PATH 的前提下检查可执行文件、Qt 插件、QML 模块、OpenSSL DLL，并启动应用进行 smoke 验证。
6. 新增以下文档：
   - `SCU_Nexus/docs/qa/manual_acceptance_checklist.md`
   - `SCU_Nexus/docs/qa/release_checklist.md`
   - `SCU_Nexus/docs/qa/optional_modules_research.md`
   - `SCU_Nexus/docs/qa/requirements_compliance_matrix.md`
7. 扩充 README，说明构建、测试、运行、打包、缓存位置和隐私注意事项。

## 8. 数据与交互流程

### 8.1 启动

```text
main
  → 注册类型与创建依赖
  → AppController.initialize
      → 打开 QueryCache
      → 初始化本地课表仓储
      → 恢复 AppSettings
  → 成功：显示 MainShell
  → 失败：显示可重试的启动错误，不静默忽略
```

### 8.2 有缓存的查询刷新

```text
进入页面
  → 读取并校验缓存
  → 展示缓存，状态 Loaded/Empty
  → 发起刷新，状态 Refreshing，内容继续可见
  → 成功：解析、写缓存、替换内容、状态 Loaded/Empty
  → 失败：保留内容和旧时间戳，恢复原状态并显示 Toast
```

### 8.3 认证与教务请求

```text
页面动作
  → ViewModel 检查登录态
  → 未登录：LoginRequired，可导航登录
  → 已登录：Service 使用共享 Cookie/网络策略
  → 响应按 HTTP、认证状态、结构与业务空结果依次判定
  → 返回统一结果，不向 QML 暴露协议细节
```

### 8.4 课表导入

```text
选择教务来源与学期
  → 登录态检查
  → 拉取并解析课程
  → 标准化旧字段与节次时间
  → 用户确认目标课表和覆盖策略
  → 仓储重新生成课程 ID 并原子替换
  → 刷新课表视图并记录不含隐私的结果摘要
```

## 9. 安全与隐私

- 不保存明文密码；
- 日志不得包含完整学号、用户名、密码、token、Cookie、验证码、完整成绩列表或完整响应正文；
- 测试样本必须使用虚构身份和脱敏数据；
- 错误提示向用户说明下一步，但不暴露 Cookie、密钥、请求签名或服务器内部细节；
- 发布包不包含调试日志、真实账号缓存、开发机绝对路径或测试凭据；
- 静态检查与单测共同约束 QML 层和日志边界。

## 10. 验证策略

### 10.1 自动化验证

每次完成一组修改后执行：

1. Windows CMake 配置与构建；
2. CTest 或现有三组测试可执行文件，并纳入新增测试；
3. QML lint，要求零错误，且本轮列出的真实布局、导入和属性覆盖警告清零；
4. `conflict_check.py`；
5. Release 构建、打包脚本和包体校验脚本；
6. 对需求矩阵中的自动化证据逐项回填结果。

测试不仅验证成功路径，还覆盖未登录、超时、损坏缓存、解析结构变化、合法空数据、写入失败、刷新失败保留和隐私日志。

### 10.2 人工验证

人工检查清单覆盖：

- 主导航、窗口最小尺寸与设置恢复；
- 登录、退出和会话过期 UI；
- 本地课表增删改、多课表切换、冲突提示；
- 教务导入、校历、考试和成绩的真实环境交互；
- 缓存优先、刷新失败保留内容、清理缓存；
- 主题、版本、空态、错误态、登录要求态；
- Windows 发布包在干净机器上的启动和基础流程。

需要校园账号或干净 Windows 的条目保留“人工待验”复选框、执行环境、日期和结果栏。在获得真实证据前不得写成已通过。
